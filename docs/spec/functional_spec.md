## 5. 핵심 사용자 흐름

### 5-1. 회원 가입과 로그인

1. 사용자는 이메일 회원가입 또는 카카오 로그인을 선택합니다.
2. 이메일 회원가입의 경우 인증 코드를 검증해야 활성화됩니다.
3. 로그인 성공 시 Access Token과 Refresh Token을 발급합니다.
4. Refresh Token은 서버에 해시로 저장해 탈취 리스크를 줄입니다.

### 5-2. 공연 조회

1. 사용자는 공연 목록에서 필터와 정렬을 사용해 공연을 찾습니다.
2. 공연 상세에서는 회차 정보, 가격 등급, 예매 정책을 확인합니다.
3. 좌석/재고 현황은 회차 단위로 조회합니다.

### 5-3. 대기열 진입

1. 특정 회차에 대기열 정책이 활성화되어 있으면 예매 전에 대기열을 거칩니다.
2. 사용자는 대기열에 진입하고 SSE로 현재 순번을 받습니다.
3. 입장 차례가 되면 Admission Token을 발급받습니다.
4. Admission Token은 10분 동안 유효합니다.

### 5-4. 좌석 hold와 주문

1. 사용자는 Admission Token을 가진 상태에서 hold를 생성할 수 있습니다.
2. 지정석은 seatIds를 선택하고, 비지정석은 sectionId와 수량을 선택합니다.
3. hold는 5분간 유효합니다.
4. hold가 생성되면 사용자는 주문을 생성하고 결제 단계로 이동합니다.

### 5-5. 결제와 확정

1. 결제는 토스페이먼츠와 직접 연동합니다.
2. 클라이언트 confirm 요청과 웹훅 모두를 처리합니다.
3. 두 요청은 도착 순서가 바뀔 수 있으므로 멱등 처리합니다.
4. 결제가 승인되면 주문 상태는 `PAID`가 됩니다.

### 5-6. 취소와 환불

1. 사용자는 공연 시작 24시간 전까지 주문을 취소할 수 있습니다.
2. 공연별로 취소 정책 override를 둘 수 있습니다.
3. 현재는 전체 취소와 전체 환불만 지원합니다.

## 6. 도메인 규칙

### 6-1. 인증 규칙

- Access Token TTL은 15분입니다.
- Refresh Token TTL은 14일입니다.
- 카카오 로그인은 OAuth2 Authorization Code 흐름을 사용합니다.

### 6-2. 대기열 규칙

- 모든 공연에 대기열을 강제하지 않습니다.
- 인기 공연에만 선택적으로 대기열을 활성화합니다.
- 대기열 통과 후 발급한 Admission Token이 있어야 hold와 주문 API를 호출할 수 있습니다.

### 6-3. 지정석 규칙

- 지정석 hold는 Redis Lua 스크립트 기반으로 원자 처리합니다.
- 최종 예매 확정은 Aurora MySQL에서 수행합니다.
- `(session_id, seat_id)` 유니크 제약으로 이중 예매를 최종 차단합니다.

### 6-4. 비지정석 규칙

- 비지정석은 구역별 가용 수량을 Redis에서 원자 차감합니다.
- 결제 완료 시 DB 재고에 반영합니다.
- DB에는 음수 재고가 되지 않도록 제약과 보정 작업을 둡니다.

### 6-5. 상태 머신

주문 상태:

- `CREATED`
- `HELD`
- `PAYMENT_PENDING`
- `PAID`
- `CANCELLED`
- `EXPIRED`
- `REFUNDED`

설명:

- `CREATED`: 주문 레코드만 생성된 상태
- `HELD`: 좌석 또는 재고 hold가 성공한 상태
- `PAYMENT_PENDING`: PG 승인 대기 또는 웹훅 대기 상태
- `PAID`: 결제 승인과 좌석 확정이 끝난 상태
- `CANCELLED`: 사용자가 취소한 상태
- `EXPIRED`: hold 만료 또는 결제 시간 초과 상태
- `REFUNDED`: 결제 취소와 환불이 완료된 상태

### 6-6. 멱등성 규칙

- 주문 생성과 결제 승인 요청은 `X-Idempotency-Key`를 사용합니다.
- 토스 웹훅도 이벤트 키 단위로 중복 처리 방지를 둡니다.

## 7. 기술 선택과 trade-off

### 7-1. 현재 저장소 구조

- 현재 저장소는 하나의 Spring Boot 애플리케이션으로 유지합니다.
- 대신 패키지 기준으로 모듈 경계를 나눠 Spring Modulith 스타일로 구현합니다.

이유:

- 학습 단계에서는 한 번에 멀티모듈과 멀티서비스를 동시에 도입하면 변경 비용이 큽니다.
- 지금은 도메인 경계와 API 계약을 먼저 안정화하는 편이 더 중요합니다.
- 대기열은 향후 별도 서비스로 분리 가능한 후보로 유지합니다.

### 7-2. 대기열을 별도 서비스 후보로 두는 이유

- 순간 트래픽 집중이 가장 큰 영역입니다.
- 코어 주문/결제와 확장 패턴이 다릅니다.
- SSE와 Redis 기반 상태 관리가 중심이라 독립 배포 가치가 큽니다.

### 7-3. 검색 엔진을 지금 도입하지 않는 이유

- 초기 범위는 목록/필터/정렬이면 충분합니다.
- 전용 검색 엔진은 운영 복잡도를 크게 올립니다.
- 카탈로그 규모와 검색 요구가 커질 때 도입을 재검토합니다.

## 8. 프론트엔드 협업 원칙

- Swagger/OpenAPI를 API 계약의 단일 기준으로 관리합니다.
- 구현 완료 기능은 슬라이스 단위로 먼저 배포하고 프론트엔드가 조기 연동할 수 있게 합니다.
- 추천 배포 순서는 `auth`, `event catalog`, `admin setup`, `queue`, `booking/payment`입니다.
- API 명세와 구현이 어긋나면 문서보다 Swagger를 우선 수정 기준으로 삼습니다.

## 10. 테스트 기준

- 동일 좌석에 1000개 동시 요청이 들어와도 최종 성공은 1건만 나와야 합니다.
- 비지정석 재고는 음수가 되면 안 됩니다.
- 결제 confirm과 웹훅은 중복, 역순 도착, 재시도 상황에서도 한 번만 반영되어야 합니다.
- hold 만료, Admission Token 만료, Redis 재시작, DB failover 시나리오를 검증합니다.
