## 1. 사용자 역할 정의

| 역할 | 설명 | 주요 권한 |
| --- | --- | --- |
| **비회원 (GUEST)** | 로그인하지 않은 사용자 | 이벤트 목록/상세 조회만 가능 |
| **회원 (USER)** | 로그인한 일반 사용자 | 대기열 진입, 좌석 선택, 예매, 알림 수신 |
| **관리자 (ADMIN)** | 이벤트 운영 관리자 | 이벤트 등록/수정, 대시보드, 입장 속도 조절 |

## 2. 이벤트 관리

### 2-1. 이벤트 목록 조회

| 항목 | 내용 |
| --- | --- |
| **사용자** | 모든 사용자 |
| **설명** | 예정된 공연/경기 목록을 조회한다 |
| **필터** | 카테고리(콘서트/스포츠/뮤지컬), 날짜 범위, 예매 상태(예정/진행중/마감) |
| **정렬** | 예매 오픈일 순 (기본), 공연일 순 |
| **페이지네이션** | 커서 기반, 한 페이지당 20건 |
| **캐싱** | 목록 데이터는 Redis 캐시 적용 (TTL 1분) |

**화면 표시 항목**: 이벤트 썸네일, 제목, 장소, 공연 일시, 예매 오픈 일시, 예매 상태 뱃지

### 2-2. 이벤트 상세 조회

| 항목 | 내용 |
| --- | --- |
| **사용자** | 모든 사용자 |
| **설명** | 특정 이벤트의 상세 정보를 조회한다 |

**화면 표시 항목**:

- 이벤트 기본 정보 (제목, 설명, 장소, 일시)
- 좌석 배치 미리보기 (구역 단위, 클릭 불가)
- 가격 등급별 금액 정보
- 예매 오픈 일시 및 남은 시간 카운트다운
- 잔여 좌석 수 (구역별 요약)
- 예매 오픈 전: "예매 오픈까지 D-n" 표시
- 예매 오픈 후: "예매하기" 버튼 → 대기열 진입

### 2-3. 이벤트 등록 (관리자)

| 항목 | 내용 |
| --- | --- |
| **사용자** | ADMIN |
| **설명** | 새로운 이벤트를 생성하고 좌석 배치를 설정한다 |

**입력 항목**:

- 기본 정보: 제목, 설명, 카테고리, 장소명, 공연 일시, 예매 오픈 일시, 썸네일 이미지 URL
- 가격 등급: 등급명, 금액 (예: VIP 150,000원 / R석 120,000원 / S석 80,000원)
- 좌석 배치: 구역명, 행 수, 열 수, 적용 가격 등급

**유효성 검사**:

- 예매 오픈 일시는 현재 시각 이후여야 한다
- 공연 일시는 예매 오픈 일시 이후여야 한다
- 좌석 배치 설정 시 행/열은 1 이상이어야 한다

**좌석 자동 생성**: 구역 설정 저장 시 개별 좌석 레코드를 Bulk Insert로 일괄 생성한다

- 예시: A구역 10행 × 20열 → 200개 좌석 자동 생성
- 각 좌석의 초기 상태: `AVAILABLE`

## 3. 대기열 시스템

### 3-1. 대기열 진입

| 항목 | 내용 |
| --- | --- |
| **사용자** | USER (로그인 필수) |
| **진입 조건** | 예매 오픈 시간 이후 + 해당 이벤트에 아직 대기열 진입하지 않은 사용자 |
| **설명** | 예매 오픈 시간에 사용자가 "예매하기" 버튼을 클릭하면 대기열에 자동 배치된다 |

**동작 흐름**:

1. 사용자가 이벤트 상세 페이지에서 "예매하기" 클릭
2. 서버가 Redis Sorted Set에 사용자를 추가 (score = 진입 시각 타임스탬프)
3. 사용자에게 현재 순번 응답 (JWT의 userId로 대기열 위치를 직접 조회)
4. 프론트엔드는 대기 화면으로 전환

**예외 처리**:

- 예매 오픈 시간 전 요청 → `400 예매 오픈 전입니다`
- 이미 대기열에 존재 → 기존 순번 정보 반환 (중복 진입 방지)
- 이미 예매 완료한 이벤트 → `409 이미 예매가 완료된 이벤트입니다`

### 3-2. 대기 순번 조회

| 항목 | 내용 |
| --- | --- |
| **사용자** | 대기열에 진입한 USER |
| **방식** | SSE (Server-Sent Events) 주 채널, 폴링 (3초 간격)은 SSE 실패 시 폴백 |
| **설명** | 현재 내 순번과 예상 대기 시간을 실시간으로 확인한다 |

**응답 정보**:

- 현재 내 순번 (예: 1,523번째)
- 전체 대기 인원
- 예상 대기 시간 (입장 속도 기반 계산)
- 현재 입장 처리 속도 (예: 분당 200명)

### 3-3. 입장 허용

| 항목 | 내용 |
| --- | --- |
| **주체** | 서버 (스케줄러) |
| **설명** | 서버 상태에 따라 대기열에서 N명씩 순차적으로 입장을 허용한다 |

**동작 흐름**:

1. 스케줄러가 주기적으로 (1~3초 간격) 대기열에서 상위 N명을 꺼냄
2. 각 사용자에게 입장 토큰(JWT) 발급 — 유효시간 10분
3. SSE 또는 폴링 응답을 통해 "입장 가능" 상태 전달
4. 프론트엔드는 입장 토큰과 함께 좌석 선택 페이지로 자동 이동

**입장 토큰**:

- JWT 기반, 페이로드에 `userId`, `eventId`, `expiredAt` 포함
- 좌석 선택/점유 API 호출 시 입장 토큰 검증 필수
- 유효시간(10분) 내 좌석 선택을 시작하지 않으면 토큰 만료 → 재진입 필요

**입장 속도 조절**:

- 기본값: 3초당 50명
- 관리자가 실시간으로 조절 가능 (서버 부하 상태에 따라)

### 3-4. 대기열 이탈 처리

| 항목 | 내용 |
| --- | --- |
| **설명** | 사용자가 의도적으로 이탈하거나, 비정상 종료 시 처리 |
- **명시적 이탈**: 사용자가 "대기 취소" 클릭 → 대기열에서 제거
- **비정상 종료**: 브라우저 종료/새로고침 시 JWT userId로 대기열 위치 복귀 가능
- **대기열 만료**: 대기열 진입 후 30분간 순번 도달하지 않으면 자동 제거

## 4. 좌석 조회 및 선택

### 4-1. 좌석맵 조회

| 항목 | 내용 |
| --- | --- |
| **사용자** | 입장 토큰을 보유한 USER |
| **설명** | 이벤트의 전체 좌석 배치를 구역 단위로 조회한다 |

**2단계 조회 구조**:

1. **구역 뷰 (Overview)**: 전체 구역 배치도 + 구역별 잔여 좌석 수/비율(색상 그라데이션)
2. **좌석 뷰 (Detail)**: 특정 구역 클릭 시 해당 구역의 개별 좌석 상태 로드

**구역 뷰 응답 데이터**:

- 구역 목록: 구역 ID, 구역명, 가격 등급, 좌표 (렌더링용), 전체 좌석 수, 잔여 좌석 수

**좌석 뷰 응답 데이터**:

- 좌석 목록: 좌석 ID, 행, 열, 상태 (`AVAILABLE` / `HELD` / `RESERVED`), 가격
- **본인이 점유 중인 좌석**은 `MY_HOLD` 상태로 별도 구분 (프레젠테이션 전용 상태, DB에는 `HELD`로만 저장됨)

**좌석 상태 → 프론트엔드 색상 매핑 (제안)**:

| 상태 | 의미 | 색상 (제안) |
| --- | --- | --- |
| `AVAILABLE` | 선택 가능 | 초록 |
| `HELD` | 다른 사용자가 점유 중 | 회색 |
| `MY_HOLD` | 내가 점유 중 (프레젠테이션 전용, DB에는 HELD) | 파랑 |
| `RESERVED` | 예매 완료 (매진) | 빨강 |

### 4-2. 좌석 실시간 상태 구독

| 항목 | 내용 |
| --- | --- |
| **사용자** | 좌석 뷰에 진입한 USER |
| **방식** | WebSocket (STOMP) |
| **설명** | 좌석 상태 변경을 실시간으로 수신한다 |

**구독 채널**: `/topic/events/{eventId}/sections/{sectionId}`

**수신 메시지 형식**:

- 좌석 ID, 변경된 상태, 변경 시각
- 한 번에 여러 좌석이 변경될 수 있음 (연석 선택/해제 시)

**구독 범위**:

- 현재 보고 있는 구역만 구독 (불필요한 메시지 최소화)
- 구역 전환 시 이전 구독 해제 → 새 구역 구독

**연결 끊김 처리**:

- 자동 재연결 (SockJS fallback)
- 재연결 시 해당 구역 좌석 전체 상태를 REST API로 한 번 동기화

### 4-3. 좌석 선택 (임시 점유 요청)

| 항목 | 내용 |
| --- | --- |
| **사용자** | 입장 토큰 보유 USER |
| **설명** | 1~4석을 선택하여 임시 점유를 요청한다 |

**입력**: 좌석 ID 목록 (1~4개)

**비즈니스 규칙**:

- 한 사용자는 하나의 이벤트에서 최대 4석까지 선택 가능
- 연석 선택 시 같은 구역, 같은 행의 연속된 좌석이어야 함 (선택적 규칙, 논의 필요)
- 이미 점유/예매된 좌석이 포함되어 있으면 전체 요청 실패

**원자성 보장**:

- 3석 선택 시, 3석 모두 AVAILABLE이면 전부 성공 / 하나라도 아니면 전부 실패
- Redis Lua Script로 원자적 처리

**성공 시**:

- 선택한 좌석 상태 → `HELD` (Redis TTL 5분)
- Hold 시점의 가격 스냅샷 저장 (이후 결제 시 가격 변동 방지)
- 점유 만료 시각 응답
- WebSocket으로 해당 구역 구독자들에게 상태 변경 브로드캐스트

**실패 시**:

- 실패 사유 응답 (이미 선택된 좌석, 최대 선택 수 초과 등)
- 프론트엔드: 실패 좌석 하이라이트 + "다른 좌석을 선택해주세요" 안내

### 4-4. 좌석 선택 해제

| 항목 | 내용 |
| --- | --- |
| **사용자** | 좌석을 점유 중인 USER |
| **설명** | 점유 중인 좌석을 자발적으로 해제한다 |
- 전체 해제 또는 부분 해제 가능 (예: 3석 중 1석만 해제)
- 해제 시 즉시 `AVAILABLE` 복구 → WebSocket 브로드캐스트
- 해제 시 Redis TTL 키 삭제 + DB 상태 업데이트

## 5. 좌석 점유 및 예매

### 5-1. 점유 타이머

| 항목 | 내용 |
| --- | --- |
| **담당** | 프론트엔드 (표시) + 백엔드 (기준 시간 제공) |
| **설명** | 좌석 점유 후 남은 시간을 카운트다운으로 표시한다 |

**동작**:

- 점유 성공 응답에 `holdExpireAt` (서버 시각 기준) 포함
- 프론트엔드는 서버 시각과 로컬 시각의 차이(offset)를 보정하여 카운트다운
- 잔여 1분 시 경고 표시 ("1분 내에 결제를 완료해주세요")
- 만료 시 자동으로 좌석 선택 해제 UI 표시 + 좌석맵 화면으로 복귀

### 5-2. 점유 자동 만료 (서버)

| 항목 | 내용 |
| --- | --- |
| **주체** | 서버 (Redis TTL + Keyspace Notification) |
| **설명** | 5분 내 결제 미완료 시 자동으로 좌석을 반환한다 |

**동작 흐름**:

1. Redis TTL 만료 → Keyspace Notification 수신 (best-effort 보조 메커니즘)
2. DB 좌석 상태를 `AVAILABLE`로 복구
3. WebSocket으로 좌석 상태 변경 브로드캐스트
4. 해당 사용자에게 "점유 시간이 만료되었습니다" 알림

> **주의**: Keyspace Notification은 best-effort 보조 메커니즘이며, 메시지 유실 가능성이 있습니다.
> 정합성 보장의 주 메커니즘은 아래 정합성 배치입니다.

**정합성 보장 (주 메커니즘)**:

- **정합성 체크 배치 (1분 간격)**: Redis에 키가 없는데 DB에 `HELD` 상태인 좌석 → `AVAILABLE`로 복구
- Keyspace Notification은 빠른 반응성을 위한 보조 수단이며, 배치가 최종 정합성을 보장

### 5-3. 예매 확정 요청

| 항목 | 내용 |
| --- | --- |
| **사용자** | 좌석을 점유 중인 USER |
| **설명** | 점유 중인 좌석에 대해 예매를 확정(결제 진행)한다 |

**사전 조건**:

- 점유 시간이 만료되지 않은 상태
- 요청한 좌석이 모두 본인이 점유 중인 좌석

**동작 흐름**:

1. 프론트엔드: 예매 정보 확인 화면 (좌석, 매수, 총 금액)
2. "결제하기" 클릭 → 예매 확정 API 호출
3. 서버: 점유 상태 재검증 → 결제 프로세스 시작 (Kafka 이벤트 발행)
4. 결제 결과에 따라 최종 상태 결정

## 6. 결제

### 6-1. 결제 처리 (모의)

| 항목 | 내용 |
| --- | --- |
| **설명** | 실제 PG 연동 대신 모의 결제 서비스로 구현한다 |

**모의 결제 서비스 동작**:

- 90% 확률 성공 / 10% 확률 실패 (장애 시뮬레이션용)
- 처리 지연: 1~3초 랜덤 (실제 PG 응답시간 모사)
- 성공 시 결제 확인 번호(paymentId) 발급

### 6-2. 결제 성공 처리

**동작 흐름**:

1. 결제 성공 이벤트 수신 (Kafka)
2. DB 좌석 상태 → `RESERVED` 확정
3. Redis 점유 키 삭제
4. 예매 내역(Booking) 레코드 생성
5. WebSocket 좌석 상태 브로드캐스트
6. 사용자에게 예매 완료 알림

**예매 내역에 포함되는 정보**:

- 예매 번호, 이벤트 정보, 좌석 목록, 총 결제 금액, 결제 확인 번호, 예매 일시

### 6-3. 결제 실패 처리 (Saga 보상 트랜잭션)

**동작 흐름**:

1. 결제 실패 이벤트 수신 (Kafka)
2. Redis 점유 키 삭제
3. DB 좌석 상태 → `AVAILABLE` 복구
4. WebSocket 좌석 상태 브로드캐스트 (좌석 재오픈)
5. 사용자에게 "결제에 실패했습니다. 좌석이 해제되었습니다." 알림
6. Dead Letter Queue로 실패 이벤트 보관 (재처리/분석용)

### 6-4. 멱등성 보장

- 모든 결제 요청에 Idempotency Key (UUID) 포함
- 동일 Idempotency Key로 중복 요청 시 이전 결과 반환 (재처리 방지)
- Idempotency Key는 Redis에 저장 (TTL 24시간)

## 9. 관리자

### 9-1. 대시보드

| 지표 | 설명 | 갱신 주기 |
| --- | --- | --- |
| 실시간 접속자 수 | WebSocket 연결 수 기준 | 실시간 |
| 대기열 크기 | 현재 대기 중인 사용자 수 | 실시간 |
| 좌석 판매율 | 전체/구역별 예매 완료 비율 | 5초 |
| 분당 예매 수 | 최근 1분간 예매 확정 건수 | 5초 |
| 평균 응답 시간 | 예매 API p50/p95/p99 | 10초 |

### 9-2. 입장 속도 조절

- 대기열에서 한 번에 입장시키는 인원 수 조절 (10~200명 범위)
- 입장 주기 조절 (1~10초 간격)
- "일시 정지" 기능: 서버 장애 시 입장을 멈추고 대기열만 유지

### 9-3. 이벤트 관리

- 이벤트 목록 조회 (관리자용, 전체 상태)
- 이벤트 수정 (공연 정보 수정, 예매 오픈 시간 변경 등)
- 이벤트 상태 변경 (예매 중지, 예매 재개)

## 10. 공통 요구사항

### 인증/인가

| 항목 | 설명 |
| --- | --- |
| 방식 | JWT (Access Token + Refresh Token) |
| Access Token 유효시간 | 30분 |
| Refresh Token 유효시간 | 14일 |
| 저장 | Access Token: 메모리(변수), Refresh Token: httpOnly Cookie |
| 토큰 갱신 | Access Token 만료 시 Refresh Token으로 자동 갱신 |

### 에러 응답 형식

```json
{
  "status": "ERROR",
  "error": {
    "code": "SEAT_ALREADY_HELD",
    "message": "이미 다른 사용자가 선택한 좌석입니다."
  },
  "timestamp": "2026-02-24T15:30:00"
}
```

### 공통 에러 코드

| 코드 | HTTP 상태 | 설명 |
| --- | --- | --- |
| `UNAUTHORIZED` | 401 | 로그인 필요 |
| `FORBIDDEN` | 403 | 권한 없음 |
| `TOKEN_EXPIRED` | 401 | 토큰 만료 |
| `INVALID_ENTRY_TOKEN` | 403 | 유효하지 않은 입장 토큰 |
| `EVENT_NOT_FOUND` | 404 | 존재하지 않는 이벤트 |
| `SEAT_ALREADY_HELD` | 409 | 이미 점유된 좌석 |
| `SEAT_ALREADY_RESERVED` | 409 | 이미 예매 완료된 좌석 |
| `MAX_SEAT_EXCEEDED` | 400 | 최대 좌석 선택 수 초과 |
| `HOLD_EXPIRED` | 410 | 점유 시간 만료 |
| `BOOKING_NOT_OPEN` | 400 | 예매 오픈 전 |
| `QUEUE_ENTRY_REQUIRED` | 403 | 대기열 미진입 |
| `ALREADY_IN_QUEUE` | 409 | 이미 대기열에 진입 |
| `DUPLICATE_BOOKING` | 409 | 이미 예매 완료한 이벤트 |
| `PAYMENT_FAILED` | 502 | 결제 실패 |
| `IDEMPOTENCY_CONFLICT` | 409 | 중복 결제 요청 |
