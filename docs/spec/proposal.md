## 📌 한 줄 소개

실시간 좌석 선점과 대기열 시스템을 갖춘, 공정하고 빠른 좌석 예매 플랫폼

---

## 🎯 기획 의도

### 우리가 해결하고 싶은 문제

콘서트, 야구 경기, 뮤지컬 — 좋아하는 공연의 티켓을 잡으려면 매번 전쟁입니다.

- 오픈 시간에 수만 명이 동시 접속하지만, 서버가 터져서 아무도 예매하지 못하는 상황
- 분명 좌석을 선택했는데 결제 직전에 "이미 선택된 좌석입니다" 알림
- 내가 몇 번째인지도 모른 채 무한 새로고침을 하는 대기 화면
- 결제 시간이 지나 풀린 좌석을 누군가 가져갔는데 알림도 없는 경험
- **티키타카(TickeTaca)**는 이런 문제를 기술적으로 해결합니다.

공정한 대기열로 순서를 보장하고, 좌석 단위의 실시간 동기화로 "내가 보는 좌석 = 실제 상태"를 보장합니다.
점유 시간이 지난 좌석은 즉시 다른 사용자에게 기회가 돌아가는, 투명하고 빠른 예매 경험을 만듭니다.

### 기대 효과

- **사용자**: 공정한 순서 보장과 실시간 좌석 현황으로 스트레스 없는 예매 경험
- **주최자**: 안정적인 트래픽 처리로 오픈 시 서버 장애 없이 매끄러운 판매 운영

## 🛠 기술 스택

### Backend

| 분류 | 기술 | 선정 이유 |
| --- | --- | --- |
| **Language** | Java 17 | LTS 안정성, Spring 생태계 호환 |
| **Framework** | Spring Boot 3.x | 빠른 프로덕션 구성, 풍부한 커뮤니티 레퍼런스 |
| **ORM** | Spring Data JPA + QueryDSL | 복잡한 좌석 조회 쿼리에 타입 세이프한 동적 쿼리 필요 |
| **DB** | MySQL 8.0 | 좌석 예매의 트랜잭션 정합성 보장 (ACID), 행 단위 락 지원 |
| **Cache** | Redis | 대기열(Sorted Set), 좌석 임시 점유(TTL), 분산 락 — 세 가지 핵심 기능에 모두 활용 |
| **Message Queue** | Apache Kafka | 예매 확정 이후 결제/알림 비동기 처리, 이벤트 순서 보장 및 재처리 가능 |
| **실시간 통신** | Spring WebSocket (STOMP) | 좌석 상태 변경을 구독 중인 사용자에게 실시간 브로드캐스트 |
| **API 문서** | SpringDoc (Swagger) | API 스펙 자동화, 프론트엔드 팀과의 소통 비용 절감 |
| **테스트** | JUnit 5 + Mockito + Testcontainers | Redis/MySQL 통합 테스트를 실제 컨테이너 환경에서 수행 |
| **부하 테스트** | k6 | 스크립트 기반으로 동시 접속 시나리오를 코드로 관리 가능 |

### Frontend

| 분류 | 기술 | 선정 이유 |
| --- | --- | --- |
| **Language** | TypeScript | 좌석 상태 등 복잡한 타입 관리에 컴파일 타임 안전성 확보 |
| **Framework** | React 18 | 좌석맵의 잦은 상태 변경에 효율적인 렌더링 (Virtual DOM) |
| **상태 관리** | Zustand 또는 TanStack Query | 서버 상태(좌석 현황)와 클라이언트 상태(UI) 분리 관리 |
| **좌석맵 렌더링** | HTML Canvas API | 5만 석 이상의 좌석을 DOM 없이 고성능으로 렌더링 |
| **실시간 통신** | SockJS + STOMP Client | Spring WebSocket과 호환, 자동 재연결 지원 |
| **스타일링** | Tailwind CSS | 빠른 UI 개발, 좌석 상태별 동적 스타일링에 유틸리티 클래스 활용 |

### 핵심 플로우

### 1) 대기열 → 입장 → 좌석 선택 플로우

```
사용자 접속
    │
    ▼
[대기열 진입] ──── Redis Sorted Set (score = timestamp)
    │
    ▼
[순번 대기] ──── SSE로 순번 업데이트 수신
    │
    ▼ (순번 도달)
[입장 토큰 발급] ──── JWT 기반, 유효시간 제한
    │
    ▼
[좌석맵 페이지 진입]
    │
    ▼
[구역별 WebSocket 구독] ──── STOMP /topic/seats/{eventId}/{sectionId}
    │
    ▼
[좌석 선택 요청]
    │
    ▼
[Redis 원자적 점유] ──── Lua Script로 가용 여부 확인 + TTL 설정을 한 번에
    │
    ├── 성공 → 5분 타이머 시작 + WebSocket 브로드캐스트 (HELD)
    └── 실패 → 클라이언트에 실패 응답 (다른 좌석 안내)
```

### 2) 좌석 점유 → 결제 → 확정 플로우

```
[좌석 점유 상태 (HELD)]
    │
    ▼
[결제 요청] ──── Idempotency Key 포함
    │
    ▼
[Kafka: 예매 이벤트 발행]
    │
    ├── [결제 서비스] ──── 모의 결제 처리
    │       │
    │       ├── 성공 → DB 좌석 상태 RESERVED 확정
    │       │          Redis 점유 키 삭제
    │       │          Kafka: 확정 이벤트 발행 → 알림 서비스 → 사용자 알림
    │       │
    │       └── 실패 → Saga 보상 트랜잭션
    │                  Redis 점유 키 삭제
    │                  DB 좌석 상태 AVAILABLE 복구
    │                  WebSocket 브로드캐스트 (AVAILABLE)
    │
    └── [5분 TTL 만료 시]
            Redis 키 자동 만료
            Keyspace Notification → DB 상태 복구
            WebSocket 브로드캐스트 (AVAILABLE)
```

### 데이터 저장 전략

| 데이터 | 저장소 | 이유 |
| --- | --- | --- |
| 이벤트, 좌석 마스터, 예매 확정 | MySQL | 트랜잭션 정합성, 영속성 보장 필요 |
| 대기열 | Redis Sorted Set | O(log N) 순위 조회, 인메모리 처리 속도 |
| 좌석 임시 점유 | Redis (TTL) | 자동 만료가 핵심, DB 부하 분산 |
| 분산 락 | Redis (Redisson) | 동시 좌석 선점 시 원자적 처리 보장 |
| 실시간 좌석 상태 변경 | Redis Pub/Sub | WebSocket 서버에 변경 이벤트 전파 |
| 예매 후 비동기 이벤트 | Kafka | 결제/알림 비동기 처리, 이벤트 순서 보장, 재처리 |

### 동시성 제어 전략 비교 (부하 테스트 대상)

프로젝트 진행 중 아래 세 가지 방식을 단계적으로 실험하고, 각각의 TPS와 정합성을 비교할 예정입니다.

| 방식 | 적용 위치 | 장점 | 단점 | 검증 포인트 |
| --- | --- | --- | --- | --- |
| **낙관적 락 (Version)** | DB (JPA @Version) | 충돌 적을 때 성능 우수 | 충돌 많으면 재시도 급증 | 인기 좌석 동시 선점 시 실패율 |
| **비관적 락 (SELECT FOR UPDATE)** | DB | 정합성 확실 | DB 커넥션 점유 시간 증가 | 동시 500명 → DB 커넥션 풀 고갈 여부 |
| **Redis Lua Script** | Redis | DB 부하 제거, 원자적 처리 | Redis ↔ DB 정합성 관리 필요 | TTL 만료와 DB 상태 불일치 복구 |

## 🗓 개발 로드맵

### Phase 1 — 기반 구축 (2주)

- 프로젝트 초기 설정 (Spring Boot, Docker Compose 환경)
- 이벤트/좌석 도메인 설계 및 CRUD API
- DB 스키마 설계 및 마이그레이션
- 기본 좌석맵 렌더링 (Canvas 기반)
- API 문서 자동화 (SpringDoc) 및 프론트엔드 연동 확인

### Phase 2 — 동시성 제어 (2주)

- Redis 기반 좌석 임시 점유(Hold) 시스템
- 분산 락 / 낙관적 락 / Lua Script 구현 및 비교
- WebSocket 기반 좌석 상태 실시간 동기화
- 연석 선택 원자적 처리
- **부하 테스트 1차**: 동일 좌석 동시 선점 시나리오 → 초과 판매 검증

### Phase 3 — 대기열 & 비동기 처리 (2주)

- Redis Sorted Set 기반 가상 대기열
- 입장 토큰 발급 및 검증
- Kafka 이벤트 파이프라인 (예매 → 결제 → 알림)
- Saga 패턴 보상 트랜잭션 (결제 실패 시 좌석 복구)
- 대기열 UI + 입장 전환 플로우 (프론트엔드)

### Phase 4 — 안정화 & 관측성 (2주)

- Prometheus + Grafana 모니터링 대시보드
- 분산 추적 (Zipkin) 연동
- **부하 테스트 2차**: 전 구간 시나리오 (대기열 → 좌석 선택 → 결제)
- 장애 시나리오 테스트 (Redis 다운, Kafka 지연, DB 커넥션 고갈)
- 트러블슈팅 문서화 및 성능 개선 기록
