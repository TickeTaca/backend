# Proposal

# TickeTaca 제안서

> 최종 수정일: 2026-03-25

---

## 1. 프로젝트 개요

TickeTaca는 인터파크, 티켓링크, 멜론티켓처럼 높은 동시 접속과 짧은 피크 트래픽을 견뎌야 하는 티켓 예매 서비스를 학습 목적으로 구현하는 프로젝트입니다.

이 프로젝트의 핵심 목표는 단순히 빠른 MVP를 만드는 것이 아니라, 대규모 트래픽 환경에서 어떤 설계와 기술 선택이 필요한지 직접 검토하고 단계적으로 구현해 보는 것입니다.

---

## 2. 해결하려는 문제

티켓 예매 서비스는 아래 문제를 동시에 다뤄야 합니다.

- 예매 오픈 시 수많은 사용자가 같은 시점에 접속하는 문제
- 동일 좌석을 여러 사용자가 동시에 선택하는 문제
- 결제 지연, 중복 결제, 웹훅 순서 뒤바뀜 같은 외부 연동 문제
- 좌석 상태와 화면 표시가 어긋나 사용자가 혼란을 겪는 문제
- 운영자가 대기열과 예매 정책을 실시간으로 제어해야 하는 문제

TickeTaca는 이 문제를 대기열, 좌석 hold, 멱등 결제, 관리자 정책 제어, 관측성 중심 아키텍처로 해결하는 것을 목표로 합니다.

---

## 3. 목표와 원칙

### 3-1. 목표

- 한국형 티켓 예매 서비스의 핵심 흐름을 재현합니다.
- 대규모 동시성 문제를 다루는 백엔드 설계를 학습합니다.
- 프론트엔드와 협업 가능한 수준의 API 계약과 문서를 유지합니다.
- 기능을 슬라이스 단위로 구현하고 검증합니다.

### 3-2. 원칙

- 빠른 MVP보다 이해 가능한 구조를 우선합니다.
- 구현 전 문서와 아키텍처 의도를 먼저 남깁니다.
- 단일 저장소에서 시작하되, 나중에 분리할 수 있는 경계를 유지합니다.
- 각 기능은 Swagger와 테스트가 함께 정리된 상태로 공개합니다.

## 5. 제안 아키텍처

### 5-1. 애플리케이션 구조

- 코어 API: Spring Boot + Spring Modulith 기반
- 대기열: 별도 서비스로 분리 가능한 Redis 중심 구조
- 현재 저장소: 학습과 점진적 구현을 위해 단일 애플리케이션 구조 유지

설명:

- 회원, 공연, 관리자, 예매, 주문, 결제는 긴밀하게 연결되어 있으므로 우선 하나의 애플리케이션에서 개발합니다.
- 대기열은 순간 확장성과 처리 패턴이 달라 별도 서비스 후보로 설계합니다.

### 5-2. 인프라

- AWS 서울 리전 단일 리전, 멀티 AZ
- Aurora MySQL
- Redis Cluster
- MSK Serverless
- ECS Fargate
- ALB

### 5-3. 관측성

- OpenTelemetry
- CloudWatch Logs
- AMP / AMG
- 주요 알람:
  - 대기열 정체
  - 좌석 hold 실패 급증
  - 결제 웹훅 실패
  - DB write latency 상승
  - Redis 오류율 상승

## 7. 핵심 설계 포인트

### 7-1. 인증

- 카카오 OAuth2 Authorization Code 로그인
- 이메일/비밀번호 회원가입과 이메일 인증
- Access Token 15분
- Refresh Token 14일
- Refresh Token 해시 저장

### 7-2. 카탈로그

- 공연 목록/상세 조회는 읽기 중심 트래픽을 고려합니다.
- Aurora Read Replica와 Redis 캐시를 전제로 설계합니다.
- 초기에는 전용 검색 엔진 없이 필터/정렬 기반 조회로 운영합니다.

### 7-3. 지정석 예매

- Redis Lua Script로 좌석 hold를 원자 처리합니다.
- Hold TTL은 5분입니다.
- 최종 예매 확정은 Aurora에서 수행하고 유니크 제약으로 중복 예매를 차단합니다.

### 7-4. 비지정석 예매

- 구역별 재고를 Redis에서 원자 차감합니다.
- 결제 완료 후 DB 재고를 반영합니다.
- 음수 재고 방지를 위해 DB 제약과 보정 작업을 둡니다.

### 7-5. 결제

- 토스페이먼츠 직접 연동
- 클라이언트 승인 요청과 웹훅을 모두 처리
- 멱등성과 중복 방지 필수

### 7-6. 비동기 이벤트

- Outbox 테이블을 기준으로 Kafka에 발행합니다.
- 소비 대상:
  - 알림
  - 감사 로그
  - 캐시 무효화
  - 티켓 발급
  - 운영 지표 적재

## 8. 프론트엔드 협업 방안

- Swagger/OpenAPI를 프론트엔드 타입 생성과 연동 기준으로 사용합니다.
- 슬라이스 단위로 API를 먼저 배포해 프론트엔드가 조기 연동할 수 있도록 합니다.
- 우선 공개 가능한 단위:
  - 인증 API
  - 공연/회차 조회 API
  - 관리자 공연/회차 설정 API
- 이후 대기열, hold, 주문, 결제 API를 순차적으로 공개합니다.
